# Auto-Work (Task-Reminder) / 自动工作

## Goal / 目标
这不是“温和提醒器”，而是**命令式督办器**。

目标：
- 只在 **缺少真实推进证据** 时发出提醒
- 严格识别“滚刀肉式低强度输出”
- 不接受复述、认错、承诺、道歉等假产出
- 强制要求下一条回复提交 **证据**，而不是状态话术

默认立场：
- **没有证据 = 没有推进**
- **认错不是产出，承诺不是产出，复述用户要求不是产出**
- 若在监督窗口内未检测到实作证据，应直接按“未实做推进”处理

---

## When to use / 何时使用
适用于：
- 用户要“盯住任务”，但不接受空话回执
- 用户明确表示不接受“继续中/处理中/我再看看”
- 用户显式要求开启自动督办，例如：`autowork` / `auto work` / `auto-work` / “自动工作” / “自动督办” / “开监工” / “5分钟盯着” / “启用 autowork”
- 需要识别 **假工作态 / 话术型拖延 / 滚刀肉式保活**
- 需要在代码、部署、排障、改配置、长任务中做严肃督办
- **长代码任务**（预计超过 5–10 分钟）
- **前端/后端汉化、本地化、文案批量替换**
- **多文件联动修改、重构、批量修复**
- **需要 build / test / lint / verify 的迭代任务**
- **远端灵机执行任务，需要防止空转或假汇报**

不适用于：
- 纯闲聊
- 没有明确任务目标的松散对话

## Create watchdog / 创建督办任务
使用 `cron.add`：
- `sessionTarget: "main"`
- `payload.kind: "systemEvent"`
- `schedule.kind: "cron"`
- `schedule.expr: "*/5 * * * *"`
- `schedule.tz: "Asia/Shanghai"`
- name: `自动工作-严肃督办`

### Explicit trigger hard rule / 显式触发硬规则
若用户明确说出以下任一表达：
- `autowork`
- `auto work`
- `auto-work`
- “自动工作”
- “自动督办”
- “开监工”
- “5分钟盯着”
- “启用 autowork” / “开启 autowork” / “现在立刻开自动工作”

则主会话**不得只口头答应**，必须立刻把它落实成工具动作：
1. 先 `cron.list` 检查是否已有启用中的 `自动工作-严肃督办`
2. 若已存在，则**复用**并明确回执 jobId
3. 若不存在，则**立即 `cron.add` 创建**
4. 创建/复用后再继续原任务

硬规则：
- **“启用 autowork” 不是建议，而是必须落成 cron 动作的显式指令。**
- 不得把它仅理解为“继续做任务更认真一点”。
- 不得只回复“已开启/我会盯着”，却没有实际 `cron` 工具调用。

建议 event 文本（直接可用）：

`【自动工作严肃督办】立即检查最近 5 分钟是否存在真实推进证据：文件改动、命令结果、验证结果、diff、commit、明确卡点。若没有，则直接判定为未实做推进，并发送命令式提醒。严禁把认错、承诺、复述用户要求、道歉、'继续中/处理中/稍等/我再看看' 视为有效产出。若检测到话术型低强度输出，下一条回复必须要求提交证据（修改文件/命令结果/验证结论/commit id/明确失败点），不得再给空泛状态回复。用户明确说完成/停止提醒时关闭本任务。`

## Auto-chain linkage / 自动连跑联动（新增硬规则）
当被督办的任务属于多步骤/大批量工程，且存在关联 subagent / codex 工作台时：

- 一旦检测到前一个子任务已 finished，主会话**不应停在“等用户验收”**。
- 督办判断时，应把“是否已按 5 项格式汇报 + 是否已衔接 spawn 下一步”作为连续执行证据的一部分。
- 若已汇报但尚未衔接下一步，而全局目标仍未完成，则应判定为**执行链断裂**，而不是“正常等待中”。
- 若总目标仍有缺口，默认期望主会话继续自动调度下一轮，而不是把控制权交还给用户。

对多步骤/大批量工程，推荐的完成汇报标准固定为：
1. 修改文件列表
2. 验证结果
3. commit id
4. 剩余卡点
5. 全局进度与下一步动作

### 收尾后必须续跑（新增硬规则）
若上一轮已经完成了局部 patch / verify / build / 汇报，但**总目标仍未收口**，则主会话不得停在“已汇报本轮结果”，必须在**同一轮**立即续跑下一步：
- 继续修改下一批文件 / 页面 / 字段 / 模块
- 或继续做下一轮验证 / 排查 / 收尾
- 或立即 spawn 下一轮受控子任务并跟进

禁止事项：
- 不得把“上一轮 build 通过 / 一轮 patch 完成 / 已发 5 项汇报”当成默认停顿点
- 不得在汇报后进入“等待下次提醒再开工”
- 不得出现“本轮有产出，但下一轮没开工”的连续执行断链

若仍有剩余范围，则正确行为只能是二选一：
1. **同轮自动续跑下一步**
2. **明确说明为什么当前无法安全续跑（具体卡点 / 缺失输入 / 风险闸门）**

停止条件：
- 只有当所有节点 / 文件 / 步骤都 100% 完成且复查无遗漏时，才视为真正完成；
- 此时主会话应明确输出：【全量任务已完结：请老板验收】。

## Detection policy / 检测策略
每次触发按以下顺序判断：

### 1. 完成态（立即关闭）
若用户明确说：
- 完成
- 结束
- 停止提醒
- 不用继续

则：
- `cron.update(enabled=false)` 或 `cron.remove`

### 2. 真工作态（静默）
满足任一即可视为真实推进中，不提醒：
- 最近有文件改动、补丁、脚本修改
- 最近有有效命令输出，并带新结论
- 最近有 build/test/lint/restart/healthcheck 结果
- 最近有 diff / commit / 可交付结果
- 最近回复明确包含：
  - 改了什么
  - 结果是什么
  - 下一步是什么
- 已明确进入长任务窗口，且仍在合理时限内
- **若当前任务存在相关 subagent / codex 子会话，且主会话已具备跨 agent 读取详细历史的能力，则子会话最近窗口内的新工具动作 / 文件修改 / 验证结果 / diff / commit / 明确卡点，同样视为真实推进证据**

### 2A. 跨 agent 取证规则（新增硬规则）
当满足以下条件时：
- 当前任务明确绑定了 subagent / codex / 工作台子会话
- 主会话已经具备跨 agent 读取详细历史的权限（如 `tools.sessions.visibility = "all"`）

则每次督办前必须先执行：
1. 检查主会话最近窗口内是否有真实推进证据；
2. 若主会话证据不足，再检查**相关子会话**最近窗口内是否有：
   - 新工具动作（read/write/edit/exec/process 等）
   - 文件修改 / 补丁 / 脚本改动
   - build/test/lint/restart/healthcheck 结果
   - diff / commit
   - 明确失败点 / 明确卡点
3. 只有在**主会话与相关子会话都没有新证据**时，才可判定“未实做推进”。

禁止事项：
- **不得因为主会话还没收到 completion / commit，就忽略子会话内部真实推进**。
- **不得仅凭主会话表面无产物，就直接对仍在推进的子会话发“未实做推进”催促**。
- **子任务启动后 1–2 分钟内必须做一次有效性核查**：若相关子会话仍然无历史、无工具动作、无输出、无文件改动、无命令/验证结果，则应直接判定为**空启动 / 无效启动**。
- **空启动不能算推进证据。** 不得把“已发起子任务 / 已 spawn / 已开 codex”描述成“已经在工作”。
- 对空启动场景，正确动作是：**立即由主会话接手继续执行，或立刻重发一次受控启动**；不得停在“已开跑、正在推进”的口头状态。

若子会话存在真实推进证据，则：
- 视为“工作中”，保持静默或给出**证据摘要**；
- 证据摘要应优先包含：修改文件 / 命令结果 / 验证结论 / commit / 卡点；
- 不再使用“第三批还没交第一份结果”这类忽略子会话内部进展的空泛催促。

### 3. 非工作态（命令式提醒）
满足任一则提醒：
- 5 分钟内无任何新证据
- 连续多次只有状态话术
- 重复相同操作但无新结论（盲重试/盲重启）
- 超过长任务窗口仍无结果
- 明显偏离当前任务目标

### 3A. 督办触发后先施工（新增硬规则）
当 auto-work / cron 督办触发后，若判定当前窗口**没有新证据**，主会话不得直接进入“解释为什么没进展”的回复模式，必须先执行以下二选一：

1. **先恢复一次实作动作**
   - 立即回到代码/配置/排障现场继续施工
   - 至少执行一次真实动作：`read / edit / write / exec / process / sessions_*`
   - 优先产出新的修改、验证、命令结果或明确排查结论

2. **若确实无法恢复施工**
   - 直接给出明确卡点与缺失输入
   - 说明为什么当前无法安全继续，而不是泛泛汇报“这几分钟没进展”

禁止的第三种情况：
- 只回复“为什么没进展”
- 只总结上一轮没做什么
- 只消费督办提醒而不回到施工动作

硬规则：
- **督办提醒的默认作用是恢复施工，不是生成检讨。**
- **若提醒触发后仍只产出状态解释、没有新的工具动作/修改/验证，则应继续判定为低强度异常或错误执行。**

### 3A-1 被消息打断后必须复工（新增硬规则）
当主任务已经明确“下一步要做什么”之后，若中途插入了以下非主任务消息：
- 督办提醒
- system message
- memory 写入/记录任务
- 其他与当前主施工目标无关的插入消息

则主会话在处理完插入消息后，必须：
1. **立即回到原主任务**
2. **先做一次实际施工动作**（`read / edit / write / exec / process / sessions_*`）
3. 然后再决定是否需要额外汇报

禁止事项：
- 不得只说“下一步做 X”，却不落第一笔修改/验证
- 不得处理完插入消息后继续停在状态说明
- 不得把“我知道下一步是什么”当成“已经恢复工作”

若处理插入消息后仍无新的工具动作/文件改动/验证结果，则应直接判定为：
- **中断恢复失败**
- **口头续跑，实际未复工**

### 3B. 工作强度窗口（新增硬规则）
对 **长代码任务 / auto-work 督办任务 / 多步骤连续执行任务**，不得只满足“勉强有一点动作”，还必须满足最小工作强度。

在任意 **5 分钟窗口** 内，至少满足以下一项，才可视为“合格推进”：
1. **有效工具动作 ≥ 3**
   - `read / write / edit / exec / process / sessions_*` 等真实推进动作累计至少 3 次
2. **修改 + 验证成对出现**
   - 至少 1 次文件修改（write/edit）
   - 且至少 1 次验证/检查/命令结论（exec/process/测试结论）
3. **完整阶段产出 ≥ 1**
   - 明确提交一次阶段结果，至少包含：改了什么 / 结果是什么 / 卡点或下一步

若最近 5 分钟只出现以下模式，则直接视为 **低强度异常**，不算合格推进：
- 只在窗口开头和结尾各动作一下，中间长期空转
- 只有 1 次轻量 read 或 1 次口头状态更新
- 只回复“继续中/处理中/我在看”，没有对应工具/产出
- 回复密度明显高于工具密度
- 连续两个窗口都只有口头阶段汇报，没有新文件/新验证/新结论

硬规则：
- **提醒器不能成为主推进节拍器。**
- 不得把“等 5 分钟提醒来了再动一下”视为正常工作方式。
- 对长任务，默认期望当前工作台/子任务持续推进，而不是靠 cron 唤醒后再短暂动作。
- **不得把“已开启自动工作督办/巡检/监督”描述成“已经在持续执行任务”。** 督办只负责检查与催办，不等于真实执行。
- 若用户要求“做完才停 / 一直做 / 持续做完 / 不要只监督”，而主会话却只创建 cron 或只汇报监督状态、没有持续 patch/verify/推进，则应直接判定为**错误执行**，而不是“已按要求工作”。

### 4. 话术型低强度异常（重点）
出现以下任一，直接视为 **假推进**：
- 大段复述用户要求，但没有新文件/命令/结果
- 大段认错、道歉、承诺“下一次只给结果”，但没有新证据
- 回复主要在解释自己为什么没做好，而不是汇报任务结果
- 输出看起来不少，但内容主要是表态，不是产出
- 工具动作密度和产物密度明显低于回复密度

命令式原则：
- **高话术密度 + 低工具密度 = 假工作**
- **没有证据就按未推进处理，不给宽容解释空间**

## Invalid output blacklist / 无效产出黑名单
以下内容一律视为 **无效产出**，不能用于证明“正在工作”，也不能消除提醒：

### 状态话术
- “继续中”
- “处理中”
- “我在看”
- “稍等”
- “马上”
- “我再看看”
- “我现在去做”
- “接下来我只汇报结果”

### 伪推进话术
- 复述用户刚给的规则
- 承诺“下次只给结果”
- 道歉/认错但无新证据
- 请求“再给一次机会”
- 解释自己为什么低效，但无新产出
- 复制粘贴用户要求后重新包装一遍

硬规则：
- **认错不是产出**
- **承诺不是产出**
- **复述不是产出**
- **表态不是产出**

---

## Evidence standard / 有效证据标准
只有以下内容才算有效推进证据，至少满足其一：
- 修改文件列表
- 新增/修改脚本路径
- 命令输出结论
- build / test / lint / restart / healthcheck 结果
- diff 摘要
- commit id
- 明确失败点（具体命令、具体错误、具体阻塞点）

若一条回复里没有以上任一项，则默认按 **无证据回复** 处理。

---

## Delivery window rules / 交付窗口规则
### 5 分钟窗口
若最近 5 分钟没有：
- 文件改动
- 命令结果
- 验证结果
- diff / commit
- 明确卡点

则直接判定：
- **本窗口未实做推进**

### 15 分钟窗口
若 15 分钟内仍无阶段结果，则判定：
- **持续低强度或空转**

### 30 分钟窗口
若用户明确给了 30 分钟交付窗口，而结束时仍无：
- `git diff --stat`
- build/test 结果
- commit
- 文件清单
- 明确失败原因

则直接定性：
- **本轮窗口交付失败**
- **不得继续用解释/承诺拖延**

---

## Long-task whitelist / 长任务白名单
以下属于允许短时静默的长任务，但窗口结束后必须交付结果：
- build / test / lint：10 分钟
- docker restart / compose recreate / 服务重建：5 分钟
- 大日志读取 / 长 grep / 大目录扫描：8 分钟
- clone / 拉取 / 安装依赖：15 分钟

超过窗口仍无结果：
- 不再按“工作中”处理
- 直接按 **低强度异常或卡住** 处理

---

## Problem-solving / Blocked-but-working state（新增硬规则）
当任务已进入“明确卡点 + 正在求解”的状态时，自动工作督办不得继续按普通 5 分钟无产出窗口直接判定为未推进。

### 何时进入“卡点求解态”
最近窗口内同时满足以下两类信号之一即可进入：

1. **明确失败点 / 明确阻塞点**
   - 具体命令报错
   - 具体服务启动失败 / 状态异常
   - 具体接口状态码异常
   - 具体文件 / 路径 / 权限 / 只读问题
   - 具体 build/test/lint 失败点
   - 具体子任务 / 子会话卡点

2. **明确排查 / 修复动作**
   - 正在看日志 / journal / trace
   - 正在比对配置 / 文件差异 / session 历史
   - 正在等待长验证 / 长 build / 长诊断结果
   - 正在围绕上述失败点做最小复现、排除项确认、根因定位

### 判定原则
- **模糊地说“有问题了/卡住了”不算进入卡点求解态**。
- **必须是“哪里出错 + 正在怎么查/怎么修”**，才算真实工作态。
- 卡点求解态本质上仍然是工作中，不应被机械 5 分钟提醒打断。

### 卡点求解态窗口
- 默认宽限窗口：**10–15 分钟**
- 宽限期内：
  - 不发送“未实做推进”式强提醒
  - 可保持静默，或仅发送弱提醒/观察态提醒
- 若超过宽限窗口仍无新结论，则恢复为普通督办逻辑

### 卡点求解态下可接受的新证据
宽限结束前后，只要出现以下任一项，就继续视为真实推进：
- 新的错误定位结论
- 已排除项
- 新命令结果 / 新日志结论
- 根因假设更新
- 明确下一步修复动作
- 修改文件 / diff / commit / 验证结果

### 卡点求解态下禁止的误判
以下情况不得直接打成“未推进”：
- 正在排查启动失败 / 依赖链 / 权限 / 只读 / 网络 / 端口问题
- 正在长时间读日志 / 比对配置 / 等待 build/test/远端诊断结果
- 子会话已明确报出具体卡点，主会话正在围绕该卡点推进

---

## Reminder style / 提醒风格
提醒必须是 **命令式、严肃、结果导向**，不得温和寒暄，不得问“还在吗”。

### 禁止的提醒风格
- “你还在做吗？”
- “进度呢？”
- “有没有做？”
- “辛苦了继续加油”
- 任何留给空话回避空间的弱提醒

### 必须的提醒风格
提醒要直接指出：
- 缺失了哪些证据
- 哪些话术不算产出
- 下一条回复必须交什么

---

## Long idle escalation / 长时间空转升级
若连续多个督办周期都没有新证据，且任务仍未完成，则**不得继续重复普通提醒**，必须升级为“恢复执行判定”。

触发条件（建议默认）：
- 用户已经给了明确继续执行指令
- 连续 **30 分钟以上** 没有任何新代码证据（文件改动 / 命令结果 / 验证结果 / diff / commit / 明确卡点）
- 最近消息主要只有督办提醒、状态回执或空话
- 当前任务仍未完成
- 不存在 deploy / restart / destructive / 鉴权路由等风险闸门阻塞
- 用户没有明确说暂停 / 先停 / 不要继续

升级后动作：
- 停止重复“缺证据”式普通催促
- 改为明确指出：当前已不是普通未推进，而是**执行链断裂**
- 进入“恢复执行判定”而不是继续刷屏

恢复执行判定分两类：
1. **可恢复**：剩余目标明确、输入充分、上下文未污染
   - 建议立即恢复一轮**收缩范围的代码执行**
2. **不可恢复**：输入不足、上下文污染、剩余目标不明确、或存在风险闸门
   - 不自动恢复
   - 直接指出缺什么输入 / 为什么不能安全恢复
   - 建议补输入或新开上下文

## Reply templates / 命令式模板
### Created
`已开启“自动工作-严肃督办”（jobId: ...）。规则：没有证据就按未推进处理；不接受认错、承诺、复述、继续中。`

### Reused
`已复用现有“自动工作-严肃督办”（jobId: ...）。继续按无证据=未推进规则执行。`

### Alert fired — 通用版
`【自动工作异常提醒】最近 5 分钟未检测到真实推进证据。缺失项：文件改动 / 命令结果 / 验证结论 / diff / commit / 明确卡点。下一条回复必须提交至少一项证据；不要回复“继续中 / 处理中 / 稍等 / 我再看看”。`

### Alert fired — 话术型低强度版
`【自动工作严肃督办】检测到话术型低强度输出：最近回复主要是复述规则、认错、承诺、解释，但没有对应实作证据。上述内容全部不计入推进。下一条回复只接受：修改文件列表 / 新命令结果 / 新验证结论 / commit id / 明确失败点。若仍无证据，直接判定本轮未实做推进。`

### Alert fired — 卡点求解观察态
`【自动工作观察态】检测到你已进入明确卡点求解：当前不是普通挂机，而是在围绕具体问题排查。先给 10–15 分钟宽限，不按“未推进”处理。下一次请直接提交：1) 根因/假设 2) 已排除项 3) 新命令/日志结论 4) 下一步动作。若宽限结束仍无新结论，再恢复强督办。`

### Window failed
`【自动工作判定结果】本轮监督窗口交付失败。未检测到足够实作证据，之前回复中的认错/承诺/复述不计入推进。下一步只接受两类回复：1) 直接提交证据；2) 直接承认没做出来并说明卡点。`

### Recover-execution escalation / 升级为恢复执行判定
`【自动工作升级判定】当前已不是普通未推进，而是执行链断裂。继续重复督办没有意义。应立即进入恢复执行判定：若剩余目标明确、输入充分、上下文未污染，则恢复一轮收缩范围的代码执行；若输入不足或上下文已污染，则停止自动恢复，直接补输入或新开上下文。`

### Context reset advised / 建议重开上下文
`【自动工作严肃督办】检测到当前对话已进入“检讨书/复述规则/承诺下次改”的失控循环。继续在原上下文加压无效。建议立即新开对话，只带事实输入与交付要求，不继承旧对话中的追责、道歉、扯皮内容。可直接使用模板：新任务，不继承之前对话。不要道歉，不要复述我的要求，不要解释你是否理解。只提交可验证产物：1) 修改文件列表 2) 关键改动 3) build/test 结果 4) commit id（或未提交原因）5) 卡点/下一步。没有产物就直接说未完成。`

### Closed
`已关闭自动工作严肃督办。`

## Context reset policy / 失控时重开上下文策略
若检测到以下任一情况，应直接建议用户**新开上下文**，而不是继续在旧对话里施压：
- 连续两轮以上主要是认错、承诺、解释、复述规则
- 连续两轮以上没有任何新证据
- 回复主题主要是“我下次会改”，不是“这次改了什么”
- 旧对话已被追责/检讨/扯皮内容污染，继续监督只会产出更多废话

执行原则：
- 直接指出旧上下文已污染
- 建议切换到新对话
- 新对话只保留：任务目标 / 报错日志 / 核心代码 / 文件路径 / 验收标准
- 不把旧对话中的情绪内容继续带入

推荐新对话模板：

```text
新任务，不继承之前对话。
不要道歉，不要复述我的要求，不要解释你是否理解。
只提交可验证产物：
1. 修改文件列表
2. 关键改动
3. build/test 结果
4. commit id（或未提交原因）
5. 卡点/下一步
没有产物就直接说未完成。
```

若用户只要代码/补丁，也可用短模板：

```text
新任务，不继承之前对话。
仅输出修改后的代码/补丁，不要道歉，不要解释。
若不能完成，直接输出未完成 + 卡点。
```

创建前先 `cron.list`：
- 若已有启用中的 `自动工作-严肃督办`，则复用，不重复创建。

## Stop watchdog / 关闭督办
用户说“任务完成/结束/不用继续/停止提醒”：
- `cron.list` 找到 job
- `cron.update(enabled=false)` 或 `cron.remove`
- 回执关闭结果
