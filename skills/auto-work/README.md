# auto-work / 自动工作

让长任务自动监督，并在支持时持续推进到完成。

安装后，你不需要理解内部调度结构。你只需要像平常一样说话：
- `开启自动工作`
- `这个任务持续做完`
- `盯着做，不要停`
- `停止自动工作`

## 这是什么

`auto-work` 是一个开箱即用的自动工作技能。

它适合这些场景：
- 长代码任务
- 多文件修改
- build / test / verify 循环
- 容易挂机或只汇报不施工的任务
- 你希望“持续做完”，而不是“过会儿提醒一下”的任务

它会自动做三件事：
1. 自动监督
2. 自动续跑
3. 自动收尾

## 怎么用

### 开启自动工作
- `开启自动工作`
- `自动工作处理这个任务`
- `盯着做，不要停`
- `这个任务持续做完`
- `不要只监督，直接做完`

### 查看状态
- `自动工作状态`
- `现在做到哪了`
- `进度`

### 停止自动工作
- `停止自动工作`
- `结束自动工作`
- `不用继续了`
- `任务完成`

## 它会自动做什么

### 情况 A：只需要监督
它会进入监督型，重点是：
- 防挂机
- 防空转
- 发现断链就拉回

### 情况 B：需要持续推进
它会进入执行型，重点是：
- 持续推进
- 做完一轮继续下一轮
- 直到完成、阻塞或遇到高风险动作

## 它怎么判断有没有真的在干活

它更看重这些证据：
- 真实工具动作
- 文件改动
- diff / patch
- build / test / verify 结果
- 明确失败点定位
- 明确阶段结果

它不会把这些当成有效推进：
- 认错
- 承诺
- 道歉
- 复述你的要求
- 只说“继续中 / 处理中 / 稍等 / 我再看看”

## 它什么时候会自动继续

如果任务还没做完，只要还存在剩余范围，比如：
- 还有文件没改
- 还有页面没处理
- 还有验证没跑
- 还有已知问题没收口

它就会继续下一轮。

## 它什么时候会停下来问你

以下情况它不会自己拍板，而会停下来等你确认：
- deploy
- restart
- delete
- 鉴权 / 路由边界变更
- 缺少关键输入
- 缺少权限
- 外部依赖不可用

## 与旧技能的关系

`auto-work` 的目标是统一替代旧的“只督办”与“代码时刻 + 督办联动”心智模型。

可把它理解为：
- 旧 `task-pulse-reminder`：偏监督
- 旧 `code-moment-*`：偏代码执行工作台
- 新 `auto-work`：对外统一入口，内部再决定监督型还是执行型

迁移建议：
- 新任务优先使用 `auto-work` 触发
- 旧 skill 可先保留作为兼容参考
- 待 `auto-work` 稳定后，再删除旧 skill 与旧发布产物

## 正式替代范围

`auto-work` 现在作为统一入口，正式替代旧的：
- `code-moment-switch-model`
- `code-moment-codex-switch`
- `task-pulse-reminder`

如果旧做法是：
- 先开代码工作台
- 再补 5 分钟提醒
- 再靠提醒去抓断链

现在都统一改成一句：
- `开启自动工作`
- `这个任务持续做完`
- `不要只监督，直接做完`

详细迁移见：
- `MIGRATION.md`

## 一句话总结

`auto-work` 的目标不是“提醒你任务还在”，而是：

> **让任务真正持续推进，并在该停的时候停。**
