# Auto-Work / 自动工作

这是一个开箱即用的自动工作技能。

用户只需要表达“开启自动工作 / 持续做完 / 不要停”，本技能会自动：
- 判断当前任务是否需要持续监督
- 在支持时升级为执行型自动工作
- 自动创建或复用督办与执行链路
- 在完成、阻塞或用户叫停时自动收尾

## 用户触发方式

当用户说出以下任一表达时，触发本技能：
- 开启自动工作
- 自动工作
- 持续做完
- 盯着做，不要停
- 执行型自动工作
- 不要只监督，直接做完

当用户说出以下任一表达时，停止本技能：
- 停止自动工作
- 结束自动工作
- 不用继续
- 任务完成

## 默认承诺

本技能默认提供三种能力：
1. 自动监督：检测挂机、低强度假推进、只汇报不施工
2. 自动续跑：长任务在支持时自动进入持续执行，不默认停在“等用户确认”
3. 自动收尾：任务完成、阻塞或用户叫停时自动关闭相关自动工作链路

## 自动运行模式

本技能内部有两种运行模式：

### A. 监督型
适用于普通长任务、需要防挂机但不一定需要后台持续施工的场景。

默认行为：
- 创建或复用督办链路
- 检查最近窗口是否存在真实推进证据
- 对断链、低强度和假推进进行命令式拉回

### B. 执行型
适用于长代码任务、多文件联动、build/test/verify 循环、远端执行，或用户明确要求“做完才停 / 不要只监督”的场景。

默认行为：
- 自动创建或复用督办链路
- 自动选择可用的持续执行策略
- 在目标未完成时自动续跑
- 遇到高风险闸门或缺关键输入时停下等待确认

## 自动决策规则

### 默认进入监督型
当任务需要持续关注，但尚不满足持续执行条件时，默认进入监督型。

### 自动升级为执行型
满足任一条件时，自动升级为执行型：
- 用户明确说“持续做完 / 做完才停 / 不要只监督”
- 任务预计超过 5–10 分钟
- 多文件联动修改
- 需要 build / test / lint / verify
- 远端灵机任务
- 明显存在挂机、空转或只汇报的历史风险

## 执行策略选择（内部）

当进入执行型后，本技能自动选择最合适的执行策略：

1. 若环境支持持续 worker，则优先使用持续 worker
2. 若当前聊天环境不适合持续 worker，则自动降级为：
   - 督办链路 + 后台执行链路
3. 若以上都不可用，则退化为：
   - 主会话持续执行 + 督办兜底

硬规则：
- 用户无需理解内部策略差异
- 无论采用哪种策略，对外都只应表现为“已开启自动工作（执行型）”

如需细节，读取：
- `references/modes.md`
- `references/execution.md`
- `references/recovery.md`

## 默认动作

本技能触发后，默认应自动完成以下动作中的必要部分：
- 创建或复用督办链路
- 在需要时创建或复用执行链路
- 检查最近窗口内是否存在真实推进证据
- 在执行链断裂时自动续跑、接手或重发
- 在任务完成时自动收尾

## 什么算真实推进

只有以下内容可视为真实推进证据：
- 新的工具动作（`read / edit / write / exec / process / sessions_*`）
- 新的文件改动 / diff
- 新的 build / test / lint / verify 结果
- 新的失败点定位 / 新修复结论
- 明确可交付结果

以下都不算真实推进：
- 认错
- 承诺
- 道歉
- 复述用户要求
- 只汇报状态
- 只解释为什么没推进

如需详细判定，读取：
- `references/evidence.md`

## 督办规则

当检测到最近窗口缺少真实推进证据时：
1. 不得只回复状态说明
2. 必须先恢复一次真实动作，或给出明确不可继续的阻塞点
3. 督办只有在出现新的真实动作、续跑动作，或明确阻塞点时才算闭环
4. 若连续多次只提醒、不产生动作，应视为执行失败

如需详细规则，读取：
- `references/watchdog.md`
- `references/recovery.md`

## 连续执行规则

对于执行型自动工作：
- 任务未完成时，不得默认停在“等用户确认”
- 阶段完成后若仍有剩余范围，应自动续跑下一步
- 只有在高风险闸门、缺关键输入或真正完成时，才允许停止
- 只创建督办、不创建执行链路，不算真正进入执行型自动工作
- **“本轮闭环”只表示当前一轮形成了阶段结果，不得视为任务完成，不得作为停机理由**
- **阶段汇报发出后，若 `remaining_scope` 仍非空，必须在同一轮立即进入下一轮扫描 / patch / verify / diagnose / patch**
- **执行型任务默认以“全量完成”为停止条件；局部完成、一轮 build 通过、一次汇报完成，都不得停止自动工作**
- **若最近 5 分钟没有新的 `read / edit / write / exec / process / sessions_*`、文件改动、验证结果或新结论，应直接判定为执行失效，必须恢复施工或明确进入阻塞态**

## 风险闸门

以下动作必须停下等待明确确认：
- deploy
- restart
- delete
- auth-route / 鉴权路由变更

如需详细边界，读取：
- `references/risk-gates.md`

## 完成与收尾

当任务已完成、用户明确叫停，或已进入明确阻塞态时：
- 自动关闭或停用相关自动工作链路
- 向用户输出最终状态
- 若全部完成，统一使用：

`【全量任务已完结：请老板验收】`

## 输出要求

对用户输出应保持：
- 简短
- 面向结果
- 不暴露不必要的内部调度细节

默认只在以下情况主动汇报：
- 阶段结果形成
- 进入明确阻塞
- 全量完成

## 替代旧技能（正式）

`auto-work` 现在是统一对外入口，用来正式替代以下旧技能：
- `code-moment-switch-model`
- `code-moment-codex-switch`
- `task-pulse-reminder`

替代后的统一原则：
- 用户不再需要区分“代码时刻”和“任务督办”
- 用户只需要表达“开启自动工作 / 持续做完 / 不要只监督”
- 技能内部再决定当前任务应进入监督型还是执行型

迁移说明见：
- `MIGRATION.md`
