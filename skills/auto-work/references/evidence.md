# Auto-Work Evidence / 自动工作证据

本文件定义 `auto-work` 判断“真实推进”所依据的证据标准。

## 1. 证据模块的目标

核心问题：
> 当前任务最近窗口内，是否真的发生了推进？

这里的“推进”是指：
- 真的做了动作
- 真的产生了结果
- 真的缩小了剩余范围
- 真的推进了任务状态

## 2. 默认判断原则

- **没有证据 = 没有推进**
- **响应不是证据**
- **承诺不是证据**
- **解释不是证据**
- **认错不是证据**

## 3. 什么算真实推进证据

满足以下任一项时，可以视为出现了真实推进证据：

### 3.1 新工具动作
- `read`
- `edit`
- `write`
- `exec`
- `process`
- `sessions_*`

### 3.2 新文件改动
- 新增文件
- 修改文件
- 删除文件
- patch / diff
- 配置更新

### 3.3 新验证结果
- build 结果
- test 结果
- lint 结果
- inspect 结果
- run / healthcheck 结果

### 3.4 新失败点定位或新修复结论
- 新失败点
- 新错误原因
- 新修复路径
- 新定位结论

### 3.5 新的可交付结果
- 可验收的修改结果
- 可交付的阶段产物
- commit / diff / 验证通过结论
- 剩余范围明显缩小

## 4. 什么不算真实推进证据

以下默认都不算：
- 认错
- 承诺
- 道歉
- 复述用户要求
- 解释为什么还没推进
- 只给状态说明
- 只说“继续中 / 处理中 / 稍等 / 我再看看”

## 5. 响应层与施工层

### 响应层
- 回复用户
- 解释情况
- 承诺继续
- 汇报计划

### 施工层
- 真实工具动作
- 文件改动
- diff / patch
- 验证结果
- 失败点定位
- 剩余范围收缩

硬规则：
- **响应层增长，不代表施工层增长。**

## 6. 低强度假推进

典型表现：
- 只有轻量 read
- 工具动作密度极低
- 响应密度高于施工密度
- 只做启动动作，不进入主循环
- 只做边缘动作，不接近目标

## 7. 工作强度窗口

在一个观察窗口内，至少满足以下之一，才可视为合格推进：
1. 出现多笔真实工具动作，且与当前目标强相关
2. 同时出现“修改 + 验证”
3. 形成一次完整阶段结果
4. 明确缩小 `remaining_scope`
5. 形成新的失败点定位并进入下一轮修复

## 8. 跨会话证据

当任务绑定了 worker / 子会话 / 后台执行链时，必须支持跨会话取证：
- 子会话中的新工具动作
- 子会话中的文件改动
- 子会话中的验证结果
- 子会话中的阶段摘要
- 子会话中的剩余范围变化

硬规则：
- **对执行型任务，不得只盯主会话文字表面。**
