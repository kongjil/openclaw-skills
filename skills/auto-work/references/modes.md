# Auto-Work Modes / 自动工作模式

本文件定义 `auto-work` 的两种运行模式，以及它们之间的自动升级、自动降级和默认选择规则。

## 1. 模式总览

`auto-work` 内部有两种模式：
1. **监督型（watch mode）**
2. **执行型（execute mode）**

区别不是“提醒语气强弱”，而是：
- 监督型：重点在检测断链、低强度和假推进
- 执行型：重点在持续推进任务直到完成、阻塞或进入风险闸门

## 2. 监督型（watch mode）

适用：
- 用户希望“盯住任务”
- 任务较长，但不一定需要后台持续执行
- 当前环境不适合稳定持续执行，但仍需要防挂机

默认行为：
- 创建或复用 watchdog / reminder 链路
- 周期性检查最近窗口内是否有真实推进证据
- 在缺少证据时触发恢复动作或要求提交阻塞点

硬规则：
- **只有监督，没有实际执行链路时，不得对外冒充“持续执行中”。**

## 3. 执行型（execute mode）

适用：
- 用户明确说“持续做完 / 做完才停 / 不要只监督”
- 长代码任务
- 多文件联动修改
- build / test / lint / verify 循环
- 远端灵机执行任务

默认行为：
- 创建或复用 watchdog 链路
- 创建或复用执行链路
- 在任务未完成时自动续跑
- 对空启动、断链、finished 但未收口等情况自动恢复

硬规则：
- **只创建督办、不创建执行链路，不算真正进入执行型。**

## 4. 默认选型规则

### 默认进入监督型
当任务需要持续关注，但尚不满足执行型条件时，默认进入监督型。

### 自动升级为执行型
满足任一条件时，自动升级为执行型：
- 用户明确要求持续做完、不要只监督
- 任务预计超过 5–10 分钟
- 需要连续多轮 patch → verify → diagnose → patch
- 明确存在剩余页面 / 剩余模块 / 剩余验证 / 剩余清扫
- 任务属于远端灵机高挂机风险场景

## 5. 升级规则

升级后的正确动作：
1. 保留或复用现有 watchdog
2. 创建或复用执行链路
3. 将当前任务状态改为执行型
4. 对外只需回执“已切换为执行型自动工作”

禁止事项：
- 不得只增强提醒文案而不补执行链路
- 不得把“我现在更认真一点”当成升级完成

## 6. 降级规则

执行型不代表必须使用同一种底层实现。

当进入执行型后，应按以下顺序自动选择策略：
1. 持续 worker
2. 后台执行链路
3. 主会话持续执行兜底

### 6A. 无 worker / 无 subagent 环境的硬规则（新增）

若当前环境下：
- `agents_list` 只返回 `main`，或
- 不存在可用的 codex / subagent worker，或
- 当前聊天环境不支持稳定后台 worker

则执行型 auto-work **不得伪装成“已有执行链”**。

此时唯一有效的执行路径应明确降级为：
- **主会话直接连续施工**（`read / edit / write / exec / process / sessions_*`）
- watchdog 只负责抓断链，不负责替代施工

硬规则：
- **没有 worker 时，主会话本身就是 execution path。**
- **不得因为“已开启 auto-work”就等待下一次提醒再施工。**
- **每次 watchdog 命中后，若最近窗口缺少新证据，主会话必须立即补至少一笔真实工具动作。**
- **若当前环境无 worker，而最近只有解释/汇报，则直接判定为执行链断裂。**

硬规则：
- **用户无需理解 run/session/thread 等内部差异。**
- **不能因为底层降级，就把执行型错误退化为“只监督”。**

## 7. 模式与完成条件的关系

### 监督型的完成
- 用户明确说完成或停止
- 当前任务已经完成
- 当前任务不再需要持续监督

### 执行型的完成
只能在以下情况结束：
- 当前总目标已经完成
- 出现明确阻塞且缺少继续推进所需输入
- 进入风险闸门并等待确认
- 用户明确要求停止

硬规则：
- **执行型不得因为“一轮 patch 完成”或“一轮汇报已发出”就默认结束。**
