# Auto-Work Watchdog / 自动工作督办

本文件定义 `auto-work` 的 watchdog 行为：
- 何时判定缺少推进证据
- 何时保持静默
- 何时必须恢复施工
- 何时判定执行链断裂
- 何时判定提醒已经闭环

## 1. 督办的定位

watchdog 不是“温和提醒器”，而是**命令式督办器**。

职责：
1. 识别最近窗口内是否存在真实推进证据
2. 识别空转、低强度假推进、只响应不施工
3. 在缺少真实推进证据时推动恢复动作
4. 在执行型场景下检查执行链是否断裂

## 2. 默认原则

- **没有证据 = 没有推进**
- **认错不是产出**
- **承诺不是产出**
- **复述用户要求不是产出**
- **提醒发出后，如果没有下一笔动作，则督办未闭环**

## 3. 何时触发督办

当满足以下任一情况时，应触发督办判断：
- 最近观察窗口缺少真实推进证据
- 连续多个窗口只有状态汇报，没有工程证据
- 出现“提醒后动一下，随后再次停工”的脉冲式施工
- 执行型任务中，worker 已 finished 但总目标未完成
- 执行链出现空启动、断链或续跑缺失

默认观察窗口一般为最近 5 分钟。

## 4. 何时保持静默

满足任一情况时，应判定为“仍在工作中”，watchdog 保持静默：
- 最近窗口内有新的工具动作
- 最近窗口内有新的文件改动 / diff
- 最近窗口内有新的 build / test / lint / verify 结果
- 最近窗口内有新的失败点定位 / 修复结论
- 关联 worker / 子会话最近窗口内存在真实推进证据

硬规则：
- **对执行型任务，不得只看主会话表面消息；必须优先检查关联 worker / 子会话证据。**

## 5. 督办前的取证顺序

1. 检查主会话最近窗口
2. 若主会话证据不足，检查关联 worker / 子会话最近窗口
3. 若都没有新证据，再判断是否发生执行链异常

## 6. 督办命中的默认动作

当 watchdog 判定最近窗口缺少真实推进证据时，必须二选一：

### A. 先恢复一次真实动作
至少做到以下之一：
- 新的 `read / edit / write / exec / process / sessions_*`
- 新的文件改动
- 新的验证动作
- 新的失败点定位

### B. 给出明确阻塞点
若确实无法继续，则必须明确说明：
- 卡在哪里
- 缺什么输入
- 为什么当前不能安全继续

## 7. 响应不能替代施工

以下都只属于“响应层”，不属于“施工层”：
- 回复提醒
- 解释原因
- 复述当前卡点
- 做进度汇报
- 承诺下一步会做什么

硬规则：
- **只响应提醒、不产生新证据，仍然视为停工。**

## 8. 督办必须闭环到下一笔动作

只有以下任一条件成立时，本次督办才算真正闭环：
1. 出现新的真实施工动作
2. 出现新的文件改动或验证结果
3. 对执行型任务，已实际发出续跑动作
4. 已明确进入阻塞态，并给出具体阻塞点

## 9. 执行型任务下的特殊职责

- 检查是否存在有效执行链路
- 检查是否发生空启动
- 检查是否发生续跑缺失
- 检查是否发生脉冲式施工

硬规则：
- **已 spawn 不等于已执行。**
- **finished 不是默认停机点。**

## 10. 完成态与停止态

当满足以下任一情况时，watchdog 应停止继续督办：
- 用户明确说完成 / 停止 / 不用继续
- 总目标已经完成
- 已进入明确阻塞态，且继续推进需要新的用户输入
- 自动工作链路已明确关闭
