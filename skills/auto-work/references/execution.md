# Auto-Work Execution / 执行型自动工作

本文件定义 `auto-work` 在执行型模式下的默认执行逻辑。

## 1. 执行型的核心目标

执行型的目标不是“提醒更严”，而是：
- 在目标未完成时持续推进
- 阶段完成后自动进入下一阶段
- 遇到异常时自动恢复执行链
- 只在完成、阻塞、风险闸门或用户叫停时停止

## 2. 执行型成立的最低条件

一个任务只有在满足以下条件时，才可算真正进入执行型：
1. 主会话承担 orchestrator 职责
2. 存在 active execution path（执行链路）
3. 存在 watchdog / 督办链路
4. 当前目标与剩余范围可被持续追踪

硬规则：
- **只有 watchdog、没有执行链路，不算执行型。**

## 3. 主会话的职责（orchestrator）

主会话负责：
1. 明确当前目标
2. 维护当前阶段与剩余范围
3. 建立或复用执行链路
4. 接收阶段结果
5. 判断是否继续下一轮
6. 在断链时接管、重发或恢复
7. 在任务完成时收尾

### 3A. 无 worker 时主会话兼任 execution path（新增）

若当前环境没有可用 worker / subagent，则主会话不只是 orchestrator，也必须直接兼任 execution path。

这意味着：
- 不能把“我已经开启自动工作”当成执行已经开始
- 不能把 watchdog / cron 提醒器当成执行器
- 不能在一轮汇报后等待用户再次推动

正确行为：
1. 主会话先做第一笔真实动作
2. 形成阶段结果后，若 `remaining_scope` 非空，主会话继续下一轮
3. 若最近窗口没有新证据，主会话按 recovery 规则直接接手，不等待外部 worker 恢复

## 4. 执行链路的职责

无论底层是什么实现，执行链路都只做一件事：
> **持续产生真实工程证据。**

默认职责：
- 读取上下文
- 做出真实动作
- 验证结果
- 发现失败点
- 继续下一轮
- 给出阶段性工程摘要

## 5. 默认执行循环

执行型任务的默认循环为：
1. `patch`
2. `verify`
3. `diagnose`
4. `patch`
5. 阶段摘要
6. 若未完成，自动进入下一轮

## 6. 阶段摘要要求

每一轮执行后，执行链路应给出简短阶段摘要，默认应包含：
1. 修改或读取了什么
2. 验证结果
3. 当前卡点
4. 下一步
5. remaining_scope

## 7. remaining_scope（剩余范围）

只要 `remaining_scope` 非空，就默认视为：
- **任务未完成**
- **应进入下一轮**

硬规则：
- **局部完成不等于全局完成。**

## 8. finished 不是默认停机点

`finished` 只表示当前一轮执行结束，不自动表示整个任务完成。

正确判断：
- 若 remaining_scope 非空：立即续跑下一轮
- 若已进入阻塞态：上报明确阻塞点
- 若已真正完成：收尾并输出最终完成消息
- **“本轮闭环”或“阶段摘要已发送”都只表示本轮形成结果，不得作为停机条件**
- **阶段摘要发出后，主会话必须立刻根据 `remaining_scope` 决定下一轮动作；若仍非空，必须同轮续跑，不得回到等待用户再次推动的状态**

## 9. 自动续跑规则

当满足以下条件时，主会话必须自动续跑：
- 总目标仍未完成
- `remaining_scope` 非空
- 当前不在风险闸门
- 当前不依赖缺失的关键用户输入
- **最近一次输出只是阶段汇报 / 局部完成 / 单轮 verify 通过，而不是全量完成**

## 10. 局部完成与全量完成

### 局部完成
以下都只算局部完成：
- 一批文件改完
- 一轮 build 通过
- 一个页面或模块处理完成
- 一次阶段汇报已发出

### 全量完成
只有满足以下条件时，才算全量完成：
- 当前总目标已完成
- 剩余范围已清空
- 无待处理验证
- 无待处理阻塞

全量完成时，统一对外输出：
- `【全量任务已完结：请老板验收】`
