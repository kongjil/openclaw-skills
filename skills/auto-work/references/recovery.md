# Auto-Work Recovery / 自动工作恢复

本文件定义 `auto-work` 在异常场景下的恢复逻辑。

## 1. 恢复模块的目标

recovery 模块的职责不是解释为什么没做好，而是：
1. 识别执行链异常
2. 判断异常属于哪一类
3. 选择正确恢复动作
4. 让任务重新回到可持续推进状态
5. 若无法恢复，则明确进入阻塞态

## 2. 恢复触发条件

当满足以下任一情况时，应进入 recovery 判断：
- worker / 执行链已创建，但最近窗口无真实动作
- worker 已 finished，但 `remaining_scope` 仍非空
- 提醒后仅短暂动作，随后再次停工
- 主会话与执行链路都缺少真实推进证据
- 阶段汇报存在，但后续没有下一轮

## 3. 恢复处理总原则

顺序：
1. 先判断是否真的异常
2. 再判断异常类型
3. 优先恢复执行链
4. 若无法恢复，则主会话接手
5. 若接手也不可行，则明确阻塞并上报

默认优先级：
1. 续跑原执行链
2. 重发受控执行链
3. 主会话接手第一笔真实动作
4. 明确阻塞并停止自动推进

## 4. 异常类型定义

recovery 模块默认处理以下四类异常：
1. 空启动
2. 执行链断裂
3. 续跑缺失
4. 脉冲式施工

## 5. 空启动（empty start）

定义：
- 已创建 worker / 已发起执行链
- 但在合理窗口内没有形成真实执行证据

正确恢复动作：
- 重发更窄、更明确的执行链
- 或主会话直接接手第一笔动作

硬规则：
- **已 spawn / 已创建，不等于已执行。**

## 6. 执行链断裂（chain break）

表现：
- 做完一轮后没有下一轮
- 只在提醒后短暂动作一下
- 只有阶段汇报，没有续跑动作

正确恢复动作：
1. 检查上一轮结果与 `remaining_scope`
2. 若仍有剩余范围，立即发起下一轮
3. 若原执行链不可复用，则创建新的受控执行链
4. 若连续多次断裂，则主会话接手

## 7. 续跑缺失（missing continuation）

定义：
- worker 已 finished
- 阶段结果存在
- 但总目标未完成，`remaining_scope` 仍非空

正确恢复动作：
1. 读取上一轮阶段摘要
2. 读取 `remaining_scope`
3. 若仍非空，则直接续跑下一轮
4. 若无法继续，则输出明确阻塞点

## 8. 脉冲式施工（pulse work）

定义：
- 只有在 watchdog / reminder 命中后才出现动作
- 动作密度只集中在提醒后短窗口
- 随后再次停工

正确恢复动作：
- 不把这类任务判定为“正常进行中”
- 要求进入完整下一轮 patch / verify / diagnose / patch
- 若多次重复，则降低对原执行链的信任并切换策略

## 9. 主会话接手的标准

当满足以下任一情况时，主会话应直接接手：
- 同一任务连续多次空启动
- 同一任务连续多次续跑缺失
- 同一任务长期呈现脉冲式施工
- 当前需要立刻建立第一笔真实动作

## 10. 明确阻塞态的进入条件

当以下情况成立时，应转入明确阻塞态：
- 缺少用户输入，无法安全继续
- 风险闸门待确认
- 缺少权限
- 外部依赖不可用，且短期无法绕过
