# 代码时刻工作流落地修复说明（2026-03-09）

本文件给主会话与代码时刻执行者一个统一约束，解决“子任务完成推送后主对话不回”“模型不可用卡住”“Phase 0/1/2 卡在等用户回复”这三类问题。

## 一、根因拆解

### Root Cause（根因）
1. 代码时刻 skill 只强调了分阶段 run，但没有把“完成推送后主会话必须先对外回复”写成**代码时刻专用硬规则**。
2. 对模型失败场景没有明确回退链路，导致 `newapi/gpt-5.3-codex` 不可用时容易停住。
3. 阶段策略文字上存在“默认确认才自动推进”的理解空间，实操中容易在 Phase 1 或 2 停等用户。

### Amplifier（放大器）
- webchat 的 subagent 是分次 run，天然容易出现“push 到了但主会话没立刻转成人类可读总结”的断点。
- 主会话若先做内部检查再回复，会让用户看到 completion payload 悬挂。

### Symptom（外在症状）
- 用户需要催“你怎么还没回复”。
- 代码时刻在模型/确认环节卡住，无法闭环到 patch+verify。

## 二、落地规则（可执行）

### 1) 完成推送后主会话接管
- completion push 到达后，主会话下一条必须是用户总结。
- 总结最少四项：完成状态、改动文件、验证结果、commit id/下一步。
- summary 发出前禁止做非必要工具调用。

### 2) 模型不可用自动回退
按顺序自动降级，不等用户：
1. 保持同任务，去掉 model 参数重试（继承主会话默认模型）。
2. 若 codex agent 不可用，改为 main 下 spawn（仍不指定 model）。
3. 若 subagent 整体不可用，主会话直做 A 流程。

### 3) Phase 0→1→2 无阻塞推进
- 默认一次 run 内尽量完成 0/1/2。
- 不因“等用户回复下一步”停在 patch/verify 之间。
- 仅 deploy/restart/destructive/鉴权路由防火墙改动强制停下确认。

## 三、与现有文件的映射

- `skills/code-moment-switch-model/SKILL.md`
  - 新增模型自动回退规则。
  - 新增 0→2 无阻塞推进硬规则。
  - 新增 completion push 后主会话接管规则。
- `AGENTS.md`
  - 在原有 handoff 章节下补充 code-moment strict handoff 子规则，确保主流程优先对外沟通。

## 四、最小验收建议（给主会话）

1. 人工模拟：发起一个代码时刻任务，要求 patch+verify。
2. 观察：subagent completion push 到达后，主会话是否第一时间发用户总结。
3. 模拟模型不可用：临时指定错误 model，确认会自动回退继续执行。
4. 检查：Phase 0→1→2 是否连续，不在“等你回复”停住。

通过以上 4 点，可基本确认本次修复是否生效。
