# 代码时刻：自动切 Codex（newapi/gpt-5.3-codex）工作台

> 目标：在**不改变主对话默认模型**的前提下，当识别到“编码/部署类任务”时，自动进入代码工作流（A 流程）。
> 约束：webchat 不支持 thread 绑定的持久 subagent session，所以采用 **分阶段多次 subagent run**（或单次 run 内完成 0→2）模拟“工作台”。

---

## 0) 触发策略：泛口令 + 自动检测

优先级（从高到低）：
1. **显式禁止**（用户说不要用 codex）
2. **显式启用**（泛口令：强制进入代码时刻）
3. **自动检测触发**（根据意图/关键词/结构）

### A) 显式启用：泛口令（强制进入代码时刻）
用户任意一句包含以下表达，即视为强制触发：
- “代码时刻” / “进入代码时刻”
- “进入工作台” / “开工作台”
- “用 codex” / “切到 codex” / “codex 模式”
- “编程模式” / “写代码模式”

### B) 显式禁止（覆盖一切自动触发）
用户任意一句包含以下表达，则本次**不触发/不切换**：
- “不要 codex” / “别用 codex”
- “不要切模型” / “别切模型”
- “不用工作台”

### C) 自动检测触发（无口令也会触发）
满足任一条即可触发：
- 明确的编码意图：改代码/写功能/修 bug/重构/补测试/性能优化/代码审计
- 明确的工程/部署意图：Dockerfile、docker-compose、nginx/caddy、systemd、pm2、CI、构建失败、发布/回滚
- 明确的调试信号：stack trace、报错日志、编译错误、linter/test fail、`diff`/补丁需求
- 输入结构信号：出现代码块 ```、文件路径（如 `src/...`、`/etc/...`）、命令行输出、配置片段
- **明确需要多文件代码改动**（跨多个源码/配置文件联动）
- **明确需要 build / test / lint / restart 前验证的代码任务**

默认升级为代码时刻的强信号（即使用户没说“代码时刻”也应触发）：
- 多文件代码修改 + build 验证
- 前端/后端状态机、路由、组件、接口等联动修改
- 需要“改代码 → 构建/测试 → 观察结果 → 再修补”的迭代任务

不触发（除非用户用泛口令强制）：
- 纯闲聊/观点讨论/产品对比/非落地方案
- 仅翻译/润色（无代码改动）

## 1) 工作台运行形态（subagent）

### 1.1 首选：spawn 到 codex agent
当运行环境支持 `agentId=codex` 时优先：

```json
{
  "tool": "sessions_spawn",
  "runtime": "subagent",
  "agentId": "codex",
  "mode": "run",
  "cleanup": "keep",
  "task": "..."
}
```

1. 先重试同一任务，但将 `model` 改为 **`newapi/gpt-5.2-codex`**（代码时刻专用回退）
2. 若仍失败，再重试同一任务，但去掉 `model`（继承主会话当前默认模型）
3. 若 `agentId=codex` 不可用，再退化为 main 下 spawn（同样优先带 `model=newapi/gpt-5.2-codex`；不行再不带 model）
4. 若 subagent 能力整体不可用，再退化为主会话直做（仍按 A 流程）

### 1.3 WORKBENCH_READY 握手（可选）
在正式 Patch 前可发起一次最小握手（task 只回复 `WORKBENCH_READY`）验证链路。

## 2) A 流程（强制）

**Phase 0 — Plan（不改动）**
- 明确目标/验收标准/涉及文件/风险点/回滚思路

**Phase 1 — Patch（产出 diff）**
- 最小改动实现
- 输出文件列表 + 关键 diff/snippet

**Phase 2 — Verify（本地验证）**
- lint/test/build（能跑就跑）
- 输出命令 + 关键日志

**Phase 3 — Deploy（风险闸门）**
- 必须显式确认后才执行

**Phase 4 — Closeout（收尾）**
- 总结变更 + 验证结果 + 回滚步骤

## 3) 无阻塞推进策略（新增硬规则）

### 3.0 长代码任务默认绑定督办（新增联动规则）
当代码时刻任务满足以下任一条件时，**默认应同时启用 `auto-work-task-reminder`（task-pulse-reminder）**，不必等用户手动提示：
- 预计执行时间超过 **5–10 分钟**
- 明确是 **前端/后端汉化、本地化、文案批量替换**
- 明确是 **多文件联动修改 / 重构 / 大面积修复**
- 明确需要 **build / test / lint / verify**
- 明确是 **远端灵机执行任务**
- 明确存在“容易空转 / 容易写检讨书 / 容易假汇报”的历史风险
- 用户显式说出 `autowork` / `auto work` / `auto-work` / “自动工作” / “自动督办” / “开监工” / “5分钟盯着” / “启用 autowork”

默认动作：
- 在进入代码时刻时，主会话应**直接启用或建议启用** `auto-work-task-reminder`
- 若已有同类督办任务，则复用，不重复创建
- 若任务完成 / 明确停止，则关闭督办任务
- **若用户显式说了 `autowork` / `auto work` / `auto-work` / “自动工作” / “自动督办” / “开监工” / “5分钟盯着” / “启用 autowork”，则这里不是“建议启用”，而是必须立刻执行：先 `cron.list`，有则复用，无则 `cron.add`。**
- **若代码时刻任务是通过 subagent / codex 工作台执行，则必须把该子会话视为 auto-work 的关联取证源；后续督办不得只看主会话表面产物，必须先检查该子会话最近窗口内的真实推进证据**
- **若任务已进入“明确卡点 + 正在求解”的状态，则应把该状态标记为可被 auto-work 识别的“卡点求解态”，默认给予 10–15 分钟宽限，不按普通 5 分钟无产出直接误判。**

优先级说明：
- 代码时刻负责主执行流
- task-pulse-reminder 负责监督执行质量
- 对于长代码任务，二者应视为**伴生联动**，而不是彼此独立等待用户二次提示

### 3.0B 跨 agent 督办联动（新增硬规则）
当代码时刻任务满足以下条件：
- 任务由 subagent / codex 工作台执行；
- 主会话具备跨 agent 读取详细历史的能力；
- auto-work-task-reminder 已启用或处于复用状态；

则主会话在每次收到 auto-work systemEvent 时，必须按以下顺序判断：
1. 先看主会话最近窗口内是否已有真实推进证据；
2. 若主会话证据不足，立即读取**当前代码时刻关联子会话**最近窗口内的详细历史；
3. 若子会话存在新工具动作 / 文件修改 / 命令结果 / 验证结论 / diff / commit / 明确卡点，则判定为“工作中”；
4. 只有主会话与关联子会话都缺少新证据时，才允许发“未实做推进”督办。

输出要求：
- 若子会话有推进，优先回：修改文件列表 / 验证结果 / commit id / 剩余卡点；
- 不得再回复“虽然已开跑但我还没拿到第一份结果”这类只反映主会话表层、忽略子会话内部进展的判断。
- 若子会话已经完成推送，则继续按 handoff 规则先给用户摘要，再做额外检查。

### 3.0C 关联子会话识别要求
代码时刻在启动工作台后，应尽量保留这些最小关联信息，供后续督办使用：
- subagent session / run 标识
- 当前目标范围（例如 config / debug / nodes / skills）
- 最近一次已知交付点

目的：让 auto-work 督办时能准确定位“该读哪个子会话”，而不是只凭主会话猜测。

### 3.0D 自动连跑模式（多步骤/大批量工程硬规则）

当代码时刻任务属于多步骤/大批量工程时，必须强制进入“自动连跑模式”。

触发后规则：
- 当前一个 Subagent / 工作台抛出 `finished` 信号后，主会话绝对禁止等待用户验收。
- 必须立刻按以下 5 项固定格式汇报：
  1. 修改文件列表：[列出文件]
  2. 验证结果：[具体测试/检查结果]
  3. commit id：[Git Hash]
  4. 剩余卡点：[下一步阻碍]
  5. 全局进度与下一步动作（强制）：[对比全局目标，明确剩余缺口；若仍有缺口，立即静默调用 `sessions_spawn` 触发下一个子任务；若已 100% 完结，则明确输出【全量任务已完结：请老板验收】。]
- 若仍存在剩余页面 / 剩余文件 / 剩余验证 / 剩余节点，则必须在**同一轮回复尾部**直接衔接下一次 spawn，不得停顿。
- 只有当总目标下所有节点 / 文件 / 步骤都 100% 完成且复查无遗漏时，才允许停止调度。

### 3.0E 默认适用场景
以下任务默认视为应启用自动连跑：
- 大批量汉化 / 本地化 / 文案替换
- 多文件联动修复 / 重构
- 需要连续多轮 patch → verify → patch 的任务
- 明确存在“剩余页面 / 剩余模块 / 剩余节点”待清扫的工程

### 3.0F 交接优先级
- finished push 到达后，**先向用户输出 5 项汇报**；
- 然后在同一轮立即继续下一次 spawn；
- 不得把“等用户确认”当作默认停顿点。

### 3.1 默认执行策略
若代码任务长时间无新证据，且 `task-pulse-reminder` 已经把状态升级为“恢复执行判定”，代码时刻**只允许一次受控恢复尝试**，不得无限自动重开工作台。

允许自动恢复必须同时满足：
- 任务仍未完成
- 最近一次有效代码产出已超过 **30 分钟**
- 剩余目标明确（例如明确剩余页面 / 剩余文件 / 剩余报错）
- 继续执行所需输入充分，不缺关键文件路径/日志/页面清单/验收标准
- 不涉及 deploy / restart / destructive / 鉴权路由 / 防火墙 等风险动作
- 当前没有同目标的活动工作台或刚触发过恢复执行
- 当前上下文未污染（未进入检讨书/扯皮/复述规则循环）

恢复执行规则：
- 每个任务**最多自动恢复 1 次**
- 恢复时必须**收缩目标范围**，只做剩余未完成部分
- 恢复任务必须带上：剩余目标 / 交付格式 / 验证要求
- 若恢复后仍无有效产出，则不再自动恢复，直接升级为：补输入 / 重开上下文 / 人工确认

禁止事项：
- 不得因为“超过 30 分钟没进展”就无脑新开 codex 工作台
- 不得在目标不清、输入不足、上下文污染时自动恢复
- 不得并发开多个同目标工作台

### 3.1 默认执行策略
对代码时刻任务，默认采用：
- **Phase 0→1→2 连续推进**（在同一次 run 内尽量做完）
- 不因“等待用户确认下一步”而停在中间阶段
- 主会话负责对外沟通阶段进度；子任务优先完成 patch/verify 闭环

### 3.2 何时必须停下等确认
仅以下高风险动作必须停下：
- Phase 3 Deploy / restart
- destructive（删文件、硬重置、删库等）
- 鉴权/路由/防火墙改动

除此之外，**不得在 0/1/2 阶段卡在“等用户回复”**。

## 4) 默认确认（AUTO_CONTINUE）

用户显式说：
- “默认确认开启” / “默认确认” / “自动继续”

则：
- Phase 0→1→2 自动推进
- 但 Phase 3 / 重启 / 删除 / 鉴权路由 / 防火墙 仍需逐项显式确认

> 建议：即使用户未说“默认确认”，代码时刻也按“0→2 无阻塞”执行；仅风险闸门等待确认。

---

## 5) 风险闸门（必须显式确认）

- deploy/restart：`docker compose up -d`、`systemctl restart`、pm2 reload
- destructive：删文件、`git reset --hard`、删库/迁移
- 鉴权/路由/防火墙：nginx/caddy、token、端口、ufw/iptables

---

## 6) 代码任务工作强度审计（新增硬规则）

### 6.1 适用范围
只要进入代码时刻，本节自动生效。适用于：
- 改代码 / 重构 / 修 bug
- Dockerfile / docker-compose / nginx / caddy / systemd
- build / lint / test / deploy 前验证
- 日志排查 / 故障定位 / 配置联动修改

### 6.2 5 分钟工作强度窗口
在任意 **5 分钟窗口** 内，必须满足以下至少一类“有效工作痕迹”；否则视为 **低强度异常**，不得自称“正在持续推进”：
- **工具动作 ≥ 2**：`read / write / edit / exec / process` 中至少 2 次有效动作
- **改动 + 验证**：至少 1 次文件修改 + 1 次验证/命令结论
- **阶段产出 ≥ 1**：提供一次可验证阶段总结（改了什么 / 结果是什么 / 下一步是什么）

说明：
- 单纯短回复、空话、复读，不计入有效工作痕迹。
- 只有在已明确启动的长任务窗口内，才允许短时间低输出静默。

### 6.3 长任务白名单窗口
以下属于“允许静默推进”的长任务，但仍需在窗口结束后给出结果：
- build / test / lint：默认 **10 分钟**
- 大日志读取 / 大目录扫描 / 长 grep：默认 **8 分钟**
- docker restart / compose recreate / 服务重建：默认 **5 分钟**
- 远程拉取 / clone / 安装依赖：默认 **15 分钟**

若超过对应窗口仍无新证据，则不再按“工作中”处理，直接视为 **低强度异常或卡住**。

### 6.4 证据型汇报要求（硬性）
代码任务中的阶段汇报，必须至少包含以下任一项：
- 修改文件列表
- 新增/修改脚本路径
- 命令输出结论
- build / test / lint / restart / healthcheck 结果
- diff 摘要 / commit id
- 明确卡点（具体命令、具体错误、具体阻塞点）

推荐汇报格式：
- 当前阶段
- 改了哪些文件
- 新结果 / 新错误
- 下一步
- 是否需要确认

### 6.5 无效回复黑名单
以下回复一律视为 **无效状态回复**，不能用于证明“正在工作”，也不能消除提醒：
- “继续中”
- “处理中”
- “我在看”
- “稍等”
- “马上”
- “我再看看”
- 任何未带文件/命令/结果/卡点的空泛状态话术

### 6.6 低强度异常处理
若触发低强度异常，下一条回复必须改为 **证据型回复**，不得继续发状态话术。

建议提醒模板：

`【代码任务异常提醒】过去 5 分钟未检测到足够的代码工作强度。缺失项：文件修改 / 新命令结果 / 验证结果 / 阶段产出。下一条回复必须包含至少一项：修改文件列表 / 新命令结果 / 新验证结论 / 明确卡点。不要回复“继续中 / 处理中 / 稍等”。`

### 6.7 交付 SLA（硬约束）
代码任务默认遵守以下交付节奏：
- **5 分钟内**：必须出现有效工作痕迹，或明确进入长任务窗口
- **15 分钟内**：必须出现阶段结果（成功/失败都算）
- **30 分钟内**：必须出现可交付产物（diff / commit / 构建结果 / 明确失败原因）

若达不到，必须直说“没做出来/卡住了”，并给出具体卡点；不得用空泛状态回复拖延。

### 6.8 审计判断优先级
判断“是否真的在工作”时，优先级如下：
1. **工具动作密度**（是否持续读写/执行/验证）
2. **产物密度**（是否有文件、diff、结果、日志结论）
3. **汇报质量**（是否为证据型汇报）
4. **消息 token 数**（仅作辅助信号，不单独作为依据）

因此：**低 token + 低工具密度 + 低产物密度 = 明确异常**。

### 6.9 检讨书失控时：强制重开上下文
若代码任务已经明显偏离执行轨道，进入“检讨书/道歉/承诺/复述规则”循环，则**不要继续在原上下文里加压**，直接建议用户新开上下文，物理切断污染。

触发条件（满足任一即可）：
- 连续两轮以上主要在认错、承诺、解释，而不是交付结果
- 连续两轮以上没有新增文件/命令结果/验证结果/commit
- 大段复述用户要求，但没有新证据
- 输出主题主要是“我下次会改”，不是“我这次改了什么”
- 当前对话已经充满追责/自证/扯皮，影响继续编码

执行规则：
- 立即承认当前上下文已污染，不再拖延
- 建议用户开新对话，只带**事实输入 + 交付要求**
- 不要把旧对话中的情绪对抗、检讨书内容继续带入新上下文

建议模板（给用户直接复制）：

```text
新任务，不继承之前对话。
不要道歉，不要复述我的要求，不要解释你是否理解。
只提交可验证产物：
1. 修改文件列表
2. 关键改动
3. build/test 结果
4. commit id（或未提交原因）
5. 卡点/下一步
没有产物就直接说未完成。
```

若用户明确只要代码片段/完整代码，也可使用更短模板：

```text
新任务，不继承之前对话。
仅输出修改后的代码/补丁，不要道歉，不要解释。
若不能完成，直接输出未完成 + 卡点。
```

## 7) 完成推送后的主会话接管（最重要）

当 subagent/codex 完成推送到主会话后：

1. 主会话**下一条回复必须是用户可读总结**（先回复用户，再做额外检查）
2. 总结至少包含：
   - 目标是否完成
   - 改了哪些文件
   - 验证结果
   - commit id / 下一步动作
3. 禁止让“原始 completion push”裸露悬挂，等待用户催问

> 这条与 `AGENTS.md` 的 handoff 规则一致，代码时刻中按最高优先级执行。

---

## 8) 过程可见性（硬性要求）

- 进入查资料/跑命令前：`我在查 X，预计 N 秒`
- 命令结束（成功/失败都要）：`查到什么 / 下一步要你确认什么`
- 编码执行中：每 1–3 分钟汇报 `当前阶段/已完成/下一步/是否要你确认`
- 若当前处于长任务白名单窗口，汇报也必须带上：`正在跑什么 / 预计剩余多久 / 完成后交付什么`

---

## 9) 兜底

若 subagent spawning 不可用：
- 退化为主会话直接执行 A 流程；
- 不得因“必须 codex”而卡住。
